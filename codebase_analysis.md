# Анализ кодовой базы Hearless: Слабые стороны и рекомендации

В данном отчете представлен подробный анализ проекта **Hearless** (монорепозиторий, состоящий из бэкенда на FastAPI, лендинга на Next.js, мобильного приложения на React Native / Expo и общего пакета shared). Были проанализированы архитектурные решения, производительность, безопасность и надежность системы.

---

## 1. Архитектурные и производительные узкие места (Performance Bottlenecks)

### 1.1. Блокирование Event Loop в FastAPI (Синхронный Whisper и API запросы)
*   **Локация:** [transcribe.py](file:///c:/Users/111/Downloads/hearless16-master/hearless16-master/backend/app/routes/transcribe.py#L8-L13)
*   **Суть проблемы:** 
    Функция обработки роута `transcribe_audio_route` объявлена как асинхронная (`async def`), однако внутри нее вызывается тяжелая синхронная функция транскрибации `transcribe_audio(contents)`:
    ```python
    @router.post("/")
    async def transcribe_audio_route(file: UploadFile = File(...), current_user: dict = Depends(get_current_user)):
        contents = await file.read()
        text = transcribe_audio(contents, language="ru") # Блокирующий вызов!
        return {"text": text}
    ```
    Локальный инференс модели Whisper (`faster_whisper`) является CPU-зависимым, а обращение к OpenAI API — сетевым блокирующим вызовом. При вызове таких функций в контексте `async def` блокируется единственный рабочий поток (Event Loop) бэкенда FastAPI. В результате все остальные входящие запросы и активные WebSocket-соединения будут «зависать» в очереди до тех пор, пока транскрипция не завершится.
*   **Рекомендация:** 
    Либо обернуть вызов в пул потоков через `await asyncio.to_thread(transcribe_audio, contents)`, либо определить роут как стандартную синхронную функцию (`def transcribe_audio_route(...)`). В последнем случае FastAPI автоматически запустит выполнение этого роута в отдельном потоке из своего встроенного пула (thread pool).

### 1.2. Неэффективное слияние аудио-чанков (Квадратичная сложность $O(N^2)$)
*   **Локация:** [main.py](file:///c:/Users/111/Downloads/hearless16-master/hearless16-master/backend/app/main.py#L128-L158) и [whisper_service.py](file:///c:/Users/111/Downloads/hearless16-master/hearless16-master/backend/app/services/whisper_service.py#L145-L183)
*   **Суть проблемы:** 
    При получении очередного 3-секундного аудио-фрагмента от мобильного устройства сервер добавляет его в список `session_chunks` и склеивает **все накопившиеся чанки с самого начала сессии**:
    ```python
    session_chunks.append(audio_bytes)
    merged_bytes = await asyncio.to_thread(merge_audio_chunks, session_chunks, first_ext)
    ```
    При длительности записи, например, 5 минут, сервер будет каждые 3 секунды считывать, декодировать через `pydub`, сохранять во временные файлы на диск и заново склеивать сотни аудиофайлов. Временная сложность склейки и последующего распознавания всей аудиозаписи растет как $O(N^2)$. Это приводит к экспоненциальному росту нагрузки на CPU, диск и память, а также к значительным задержкам ответа (lag) к концу сессии.
*   **Рекомендация:** 
    Перейти на потоковое вещание аудио (например, передача сырого PCM по WebSocket) с инкрементальным распознаванием на стороне Whisper (без склеивания старых файлов), либо хранить открытый буфер в оперативной памяти бэкенда без постоянной записи чанков на диск через временные файлы.

### 1.3. Лаги и прерывания при записи звука на мобильном (Audio Gap)
*   **Локация:** [useStreamingRecording.ts](file:///c:/Users/111/Downloads/hearless16-master/hearless16-master/mobile/src/hooks/useStreamingRecording.ts#L90-L125)
*   **Суть проблемы:** 
    На стороне мобильного приложения каждые 3 секунды срабатывает таймер, который полностью останавливает текущую запись, сохраняет её, отправляет и создает новую запись:
    ```typescript
    await oldRec.stopAndUnloadAsync();
    // ... инициализация новой записи ...
    const { recording } = await Audio.Recording.createAsync(...);
    ```
    Инициализация аудио-драйвера и файловой системы на смартфонах занимает от 100 до 350 мс. Во время остановки и старта записи микрофон физически не записывает звук. Это приводит к постоянным «выпаданиям» звука (audio gaps) каждые 3 секунды. В результате слоги или слова, произнесенные в моменты переключения, теряются, что сильно снижает качество распознавания речи.
*   **Рекомендация:** 
    Использовать библиотеки для потокового сбора аудио буфера без остановки записи (например, `react-native-live-audio-stream` или прямую запись в буфер PCM).

### 1.4. Избыточное использование диска и сети при практике жестов
*   **Локация:** [GesturePracticeScreen.tsx](file:///c:/Users/111/Downloads/hearless16-master/hearless16-master/mobile/src/screens/GesturePracticeScreen.tsx#L79-L106)
*   **Суть проблемы:** 
    Для распознавания жестов мобильное приложение каждые 300 мс вызывает `takePictureAsync` с фронтальной камеры:
    ```typescript
    const photo = await cameraRef.current.takePictureAsync({ base64: true, quality: 0.4 });
    ```
    Этот метод делает полноценный снимок, записывает его во временный файл на флеш-память устройства, перекодирует в Base64 и удаляет. Это вызывает колоссальную нагрузку на дисковую подсистему телефона, приводит к перегреву, быстрой разрядке батареи и подергиваниям (stuttering) интерфейса. Более того, отправка картинок каждые 300 мс расходует около 200–300 КБ/сек интернет-трафика пользователя.
*   **Рекомендация:** 
    Использовать Frame Processors (например, в `react-native-vision-camera`), которые позволяют считывать кадры напрямую из видеопотока в видеопамяти без сохранения файлов, а также уменьшить разрешение кадра (например, до 224x224 пикселей) перед отправкой на сервер.

## 2. Уязвимости безопасности (Security Vulnerabilities)

### 2.1. Утечка приватных API-ключей в клиентский бандл (Next.js)
*   **Локация:** [page.tsx](file:///c:/Users/111/Downloads/hearless16-master/hearless16-master/landing/app/subtitles/page.tsx#L94) и [.env](file:///c:/Users/111/Downloads/hearless16-master/hearless16-master/landing/.env#L8)
*   **Суть проблемы:** 
    Ключ Gemini API зашит прямо во фронтенд-коде и экспортируется в публичную переменную окружения с префиксом `NEXT_PUBLIC_`:
    ```typescript
    const apiKey = process.env.NEXT_PUBLIC_GEMINI_API_KEY || "AIzaSyBH3qC3cisbCHtH8CTZKmkg1DLDvdEOfEg";
    ```
    Любая переменная с префиксом `NEXT_PUBLIC_` компилируется в финальный JavaScript-бандл, отправляемый в браузер пользователя. Любой злоумышленник или сторонний пользователь может легко извлечь этот ключ из консоли/исходного кода страницы и совершать запросы к API за ваш счет, что может привести к исчерпанию квот или огромным счетам.
*   **Рекомендация:** 
    Никогда не вызывать сторонние AI API напрямую из браузера. Все запросы к Gemini / OpenAI должны проходить через ваш бэкенд (FastAPI) или через внутренние Next.js API Routes (Serverless функции), где ключи хранятся на сервере в безопасных `.env`-переменных без префикса `NEXT_PUBLIC_`.

### 2.2. Публикация Supabase Service Role Key в лендинге
*   **Локация:** [.env](file:///c:/Users/111/Downloads/hearless16-master/hearless16-master/landing/.env#L5)
*   **Суть проблемы:** 
    В файле конфигурации `.env` лендинга присутствует `SUPABASE_SERVICE_KEY` (начинающийся на `sb_secret_`). Этот ключ обладает правами администратора (`service_role`) и обходит все правила защиты таблиц (Row Level Security - RLS) в Supabase. Если данный файл попадет в систему контроля версий (Git) или будет случайно скомпилирован в сборку, злоумышленники получат полный доступ к чтению, изменению и удалению всех данных в вашей базе.
*   **Рекомендация:** 
    На стороне клиента (лендинг, мобильное приложение) должен использоваться **исключительно** публичный `anon_key` (`NEXT_PUBLIC_SUPABASE_ANON_KEY`). Секретный service key должен храниться только на бэкенде в жестко защищенной среде.

### 2.3. Отсутствие валидации Base64 данных при распознавании жестов
*   **Локация:** [gestures.py](file:///c:/Users/111/Downloads/hearless16-master/hearless16-master/backend/app/routes/gestures.py#L30-L34)
*   **Суть проблемы:** 
    Декодирование картинки происходит без предварительной проверки формата и обработки исключений:
    ```python
    @router.post("/recognize")
    async def recognize(data: GestureRecognizeRequest):
        frame = base64.b64decode(data.image) # Опасный вызов
        result = recognize_gesture(frame, data.target_gesture)
        return result
    ```
    Если клиент отправит некорректную строку (например, содержащую префикс `data:image/jpeg;base64,`), `base64.b64decode` выбросит ошибку `binascii.Error`, что приведет к падению обработчика с кодом 500. В худшем случае отправка огромных некорректных строк может вызвать перегрузку памяти или отказ в обслуживании (DoS).
*   **Рекомендация:** 
    Добавить валидацию входной строки: отсекать заголовок Data URL схемы, если он присутствует, и обернуть вызов декодера в блок `try-except` с возвратом HTTP-статуса `400 Bad Request`.

### 2.4. Потенциальная уязвимость Cross-Site WebSocket Hijacking (CSWSH)
*   **Локация:** [main.py](file:///c:/Users/111/Downloads/hearless16-master/hearless16-master/backend/app/main.py#L35-L45)
*   **Суть проблемы:** 
    CORS-мидлвар в FastAPI не защищает WebSocket-соединения автоматически. В роуте `@app.websocket("/ws/transcribe")` отсутствует проверка заголовка `Origin`. Это позволяет сторонним сайтам открывать веб-сокеты к вашему бэкенду от имени авторизованного пользователя, если токен сессии будет перехвачен или передан через параметры.
*   **Рекомендация:** 
    Внедрить явную проверку заголовка `Origin` при установлении WebSocket-соединения (handshake) на соответствие белому списку доверенных доменов.

## 3. Функциональные несоответствия и логические ошибки (UX & Reliability)

### 3.1. Заглушка (эмуляция) вместо реального распознавания жестов
*   **Локация:** [signflow_model.py](file:///c:/Users/111/Downloads/hearless16-master/hearless16-master/backend/app/signflow_model.py#L56-L57)
*   **Суть проблемы:** 
    Функция `recognize_gesture` полностью игнорирует переданные байты изображения и вызывает эмулятор, возвращающий случайные значения точности совпадения жеста:
    ```python
    def recognize_gesture(frame_data: bytes, target_gesture: str | None = None) -> dict:
        return recognize_emulate(target_gesture) # Возвращает случайные числа
    ```
    Несмотря на то, что мобильное приложение тратит батарею, интернет-трафик и дисковые ресурсы на отправку кадров, бэкенд возвращает псевдослучайные результаты. Это вводит пользователя в заблуждение, так как реальное распознавание жестов отсутствует.
*   **Рекомендация:** 
    Реализовать интеграцию с легковесной библиотекой компьютерного зрения (например, Google MediaPipe Hands) для извлечения ключевых точек руки и классификации жеста на сервере, либо перенести инференс непосредственно на мобильное устройство (например, через MediaPipe в React Native).

### 3.2. Раскрытие системных ошибок авторизации клиенту (Information Disclosure)
*   **Локация:** [main.py](file:///c:/Users/111/Downloads/hearless16-master/hearless16-master/backend/app/main.py#L58-L64) и [dependencies.py](file:///c:/Users/111/Downloads/hearless16-master/hearless16-master/backend/app/dependencies.py#L23-L26)
*   **Суть проблемы:** 
    При возникновении непредвиденных исключений во время проверки токена через Supabase, приложение возвращает клиенту детализированный текст ошибки:
    ```python
    except Exception as e:
        raise HTTPException(
            status_code=401, detail=f"Ошибка авторизации: {str(e)}"
        )
    ```
    Это может раскрыть внутреннее устройство бэкенда (версии библиотек, структуру БД, внутренние сетевые адреса Supabase) злоумышленникам.
*   **Рекомендация:** 
    Логировать детальную ошибку на сервере (например, через стандартную библиотеку `logging`), а клиенту возвращать обобщенное сообщение, например: `"Ошибка авторизации. Обратитесь в поддержку или попробуйте позже"`.

### 3.3. Различие в путях роутера API в мобильном приложении
*   **Локация:** [whisper.ts](file:///c:/Users/111/Downloads/hearless16-master/hearless16-master/mobile/src/services/whisper.ts#L22) и [transcribe.py](file:///c:/Users/111/Downloads/hearless16-master/hearless16-master/backend/app/routes/transcribe.py#L5)
*   **Суть проблемы:** 
    В бэкенде роут определен с префиксом `prefix="/transcribe"`, а сам обработчик висит на `@router.post("/")`. Таким образом, точный адрес эндпоинта — `/transcribe/` (с закрывающим слэшем).
    В мобильном приложении запрос отправляется без слэша:
    ```typescript
    const response = await axios.post(`${API_URL}/transcribe`, formData, ...);
    ```
    Это может приводить к лишнему перенаправлению 307 (Temporary Redirect) со стороны FastAPI или к ошибке `404 Not Found` (или `405 Method Not Allowed`), если клиентская библиотека некорректно обрабатывает редирект для POST-запросов (например, сбрасывает тело запроса или метод на GET).
*   **Рекомендация:** 
    Привести адреса к единому стандарту. Рекомендуется использовать пути без принудительного закрывающего слэша в FastAPI (`redirect_slashes=False`) либо указывать точные адреса `/transcribe/` на стороне клиента.
