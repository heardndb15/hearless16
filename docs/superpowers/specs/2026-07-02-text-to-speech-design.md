# Text-to-Speech — Hearless

**Date:** 2026-07-02
**Scope:** Новый раздел «Текст → Речь» — озвучивание введённого текста для общения со слышащими собеседниками

## Назначение

Зеркальная функция к уже существующей `/text-to-sign`, только в обратную сторону: глухой/слабослышащий пользователь печатает фразу, устройство озвучивает её вслух для слышащего собеседника.

## Языки

Казахский, русский, английский — тот же набор, что уже используется в проекте (регистрация: `kk`/`ru`, text-to-sign: kk/ru/en).

## Движок озвучки

Облачный TTS — **OpenAI `tts-1`**, а не браузерный `speechSynthesis`.

Причина: в проекте уже был прецедент с распознаванием речи — браузерный Web Speech API был полностью отключён для казахского (`kk`) во всех точках входа (см. решение "Kazakh transcription is FreedomSpeech-only everywhere") из-за ненадёжной поддержки `kk-KZ` в браузерах. Тот же риск актуален и для синтеза речи: браузерные `kk-KZ` голоса часто отсутствуют или звучат с чужим акцентом. Облачный TTS даёт одинаковое качество на всех языках и устройствах.

## Архитектура

Без изменений в Python-бэкенде (`backend/`). Используется тот же паттерн, что и у `landing/app/api/transcribe/route.ts`: самодостаточный Next.js API route, не зависящий от того, поднят ли FastAPI-бэкенд.

- **`landing/app/api/tts/route.ts`** (новый) — `POST { text: string, language: "kk"|"ru"|"en" }` → вызывает OpenAI `audio.speech.create({ model: "tts-1", voice, input: text })` → возвращает аудио (mp3) как бинарный ответ.
- **`landing/app/text-to-speech/page.tsx`** (новый) — публичная страница, `"use client"`, по образцу `app/text-to-sign/page.tsx`.

### Голоса

Один голос OpenAI TTS на каждый язык (конкретный подбор — на этапе реализации, проверить на слух все три языка перед сдачей). Модель `tts-1` не принимает параметр языка явно — язык определяется по входному тексту, поэтому конкретная фраза на демо-кнопках должна быть на выбранном языке.

## UI

По образцу `/text-to-sign`:

- Заголовок «Текст → Речь», подзаголовок с описанием
- Переключатель языка (Қаз / Рус / Eng) — влияет на набор демо-фраз и на голос
- `<textarea>` для ввода текста + демо-фразы (кнопки-чипы) под ним
- Кнопка «Озвучить →» — неактивна при пустом тексте
- После успешной генерации: `<audio controls autoPlay>` с полученным mp3
- Блок «Как это работает» (3 шага), как в text-to-sign

## Обработка ошибок

- `OPENAI_API_KEY` не настроен → route возвращает ошибку → страница показывает текст ошибки под кнопкой, ничего не роняет
- Сетевая ошибка / таймаут OpenAI → аналогично, текст ошибки, текст пользователя не теряется (остаётся в textarea)
- Пустой текст → кнопка отключена, запрос не отправляется

## Навигация

Новая карточка «Текст → Речь» добавляется в список `FEATURES` на `/features` (`landing/app/features/page.tsx`), в том же формате, что и остальные 4 карточки — **без ссылки** (у существующих карточек features сейчас тоже нет ссылок на страницы фич, это сознательно не меняется в рамках этой задачи).

## Предварительное условие (блокер до реализации)

`OPENAI_API_KEY` сейчас:
- в `backend/.env` — placeholder (`sk-your-openai-key`), не настоящий ключ
- в `landing/.env.local` — отсутствует вовсе

Без настоящего ключа TTS не заработает ни локально, ни на Vercel. Реальный ключ должен быть получен и добавлен пользователем — сгенерировать его нельзя.

## Тестирование

Ручная проверка (страница демо-стиля, как text-to-sign — без автотестов):
1. Запустить `landing` локально, открыть `/text-to-speech`
2. Проверить озвучку демо-фраз на всех трёх языках — прослушать, убедиться что звучит разборчиво и на нужном языке
3. Проверить пустой ввод (кнопка неактивна)
4. Проверить поведение при отсутствующем/неверном `OPENAI_API_KEY` (ошибка показывается, приложение не падает)
