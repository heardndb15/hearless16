# Сайт-вайд dark-mode safety sweep (Design Spec)

**Date:** 2026-07-14
**Scope:** `landing/app/globals.css`, `landing/app/layout.tsx`, `landing/components/Providers.tsx`, `landing/components/Header.tsx`, и ~34 файла с захардкоженными цветами (полный список — раздел «Файлы в скоупе»)
**Предшествует:** `2026-07-14-landing-premium-redesign-design.md` (лендинг-редизайн строится поверх этого спека)

## Проблема

Пользователь хочет глобальный переключатель light/dark (действует на весь сайт, не только на `/`). Но сайт красит себя смесью CSS-переменных (`var(--text)`, `var(--bg)` и т.д.) и захардкоженных литералов (`"#FFFFFF"`, `"white"`, `rgba(0, 0, 0,0.12)`) в inline-стилях. Конкретно подтверждено на `app/pricing/page.tsx:259` и `:309`: текст красится в `var(--text)` (токен, будет реагировать на тему), а лежит на захардкоженном `background: "white"` (не реагирует). Если `--text` в dark-теме станет светлым (что и есть цель) — этот текст станет невидимым: белый на белом фоне, который дал сбой не сменился на тёмный. Это системный паттерн, не единичный случай — grep находит ~220 захардкоженных цветовых литералов в ~35 файлах по всему `landing/`.

Включать глобальный переключатель до этой правки — значит осознанно ронять читаемость в разных местах сайта.

## Цель

Сделать так, чтобы переключение темы на `<html data-theme="light"|"dark">` было безопасно **в любом месте сайта**: каждый элемент, чей цвет предполагает связь с темой (фон карточки, текст, бордер, тень), задаётся через CSS-переменную, а не литерал — так, чтобы фон и текст всегда меняются согласованно (парой), а не по отдельности.

**Не цель:** визуальный редизайн этих страниц. Значения токенов в этом спеке подбираются так, чтобы **light-тема выглядела визуально идентично сегодняшнему сайту** (это рефактор безопасности, не редизайн). Премиум-минималистичная палитра — предмет следующего спека (`landing-premium-redesign`), который меняет сами *значения* токенов; этот спек только гарантирует, что смена значений будет безопасной везде.

## Инфраструктура темы

Создаётся здесь (используется в обоих спеках):

- `landing/lib/ThemeContext.tsx` — React-контекст по образцу существующего `LanguageContext.tsx`: `{ theme: "light" | "dark", setTheme }`.
- `landing/components/Providers.tsx` — оборачивается в `ThemeProvider` (рядом с `LanguageProvider`).
- `landing/app/layout.tsx` — блокирующий инлайн-скрипт в `<head>` (до первой отрисовки): читает `localStorage.getItem("hearless-theme")`, иначе `window.matchMedia("(prefers-color-scheme: dark)")`, ставит `document.documentElement.dataset.theme`.
- Переключатель (иконка солнце/луна) в `Header.tsx`, сохраняет выбор в `localStorage`.

## Механическая замена литералов

Каждый файл в скоупе построчно грепается на цветовые литералы и заменяется на **существующий** (из таблицы выше) токен — не на новый литерал. Таблица соответствий (по смыслу использования, не только по значению — см. правило ниже):

| Литерал | Контекст | Заменить на |
|---|---|---|
| `"#FFFFFF"` / `"white"` | Фон карточки/секции/дропдауна | `"var(--bgCard)"` |
| `"#FFFFFF"` / `"white"` | Цвет текста/иконки на цветном (accent/gradient) фоне | `"var(--white)"` |
| `rgba(0, 0, 0,0.06..0.2)` | Бордер | `"var(--border)"` |
| `rgba(0, 0, 0,0.06..0.09)` (box-shadow) | Обычная тень карточки | `"var(--shadow)"` |
| `rgba(0, 0, 0,0.25..0.3)` (box-shadow) | Усиленная тень (выделенный тариф и т.п.) | `"var(--shadowStrong)"` |
| `#94a3b8` | Приглушённый/неактивный текст | `"var(--textMuted)"` |
| `#DC2626` | Текст ошибки | `"var(--danger)"` |
| `rgba(12,74,110,0.35)` | Затемнение фона под мобильным меню/модалкой | `"var(--overlay)"` |
| `rgba(255,255,255,0.1..0.35)` на фоне `var(--accent)`/`var(--gradient-accent)` | Полупрозрачная рамка/фон **поверх акцентного фона** | не трогать — авто-безопасно (акцент не становится светлым в dark, контраст сохраняется) |

**Правило для неоднозначных случаев:** если один и тот же литерал используется в разных ролях в разных местах файла (например, `"white"` как фон карточки в одном месте и как цвет текста на кнопке в другом) — смотреть на роль в конкретном месте, не менять все вхождения одинаково.

## Проверка

- `npx tsc --noEmit` и `npm run build` в `landing/` — должны пройти без ошибок после каждого файла/группы файлов.
- Grep-проверка на остаток литералов: `rg "#[0-9A-Fa-f]{6}|rgba\(0, 0, 0,|rgba\(255,255,255,|\"white\"|\"#FFFFFF\"" landing/app landing/components --glob '*.tsx'` — после свипа не должен находить ничего вне явных исключений (community, sign-language/practice) и вне намеренно оставленных on-accent полупрозрачных случаев.
- Ручная проверка: переключить тему на `/`, пройтись по 4-5 репрезентативным немигрированным до этого страницам (`/pricing`, `/about`, `/dashboard`, `/login`, `/features`) — все читаемо в обеих темах, включая **сравнение тарифов** и **FAQ** на `/pricing` (конкретное место, где был найден баг).
- Это в основном рефакторинг цвета — часть проверки неизбежно ручная (визуальный обход), автоматика не заменяет её.

## Вне скоупа

- Любые изменения структуры, контента, копирайта — только замена литералов на токены. Ноль изменений текста/вёрстки/логики.
- Премиум-минималистичная палитра — значения токенов здесь совпадают с сегодняшними; их смена на новую палитру происходит в `landing-premium-redesign` спеке, который выполняется после этого.
- Geist-шрифт, IA-изменения главной страницы — там же, не здесь.
